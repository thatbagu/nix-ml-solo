# EC2 VM

## What runs on EC2

The EC2 instance runs NixOS and hosts:

- MLflow tracking server (accessible locally via SSH tunnel)
- Your project files (synced via mutagen)
- The devenv shell (for `train-on-ec2`)

## Default configuration

Set in `devenv.nix` and passed to Terraform:

```nix
env = {
  TF_VAR_instance_type = "t3.small";    # default
  TF_VAR_aws_region    = "us-east-1";
};
```

## Adding packages and services

Edit `TF_VAR_ec2_extra_nix_config` in `devenv.nix`:

```nix
env.TF_VAR_ec2_extra_nix_config = ''
  environment.systemPackages = with pkgs; [ htop ripgrep nvtop ];
  services.prometheus.enable = true;
'';
```

This NixOS module expression is merged into the base EC2 configuration. Then apply the change:

```sh
tf-plan && tf-apply    # replace instance (if module changes require it)
# or:
nixos-rebuild          # push config to existing instance without replacing it
```

Use `nixos-rebuild` for additive changes (new packages, services). Use `tf-apply` for changes that require instance replacement (different AMI, new storage).

## `nixos-rebuild`

Pushes a NixOS config change to the running EC2 instance via SSH without destroying and recreating it:

```sh
nixos-rebuild
```

This:
1. Copies `.devenv-configs/nixos-config.nix` to the EC2 instance
2. Runs `nixos-rebuild switch` remotely

The config file is generated by Terraform from your `devenv.nix` values. If you change `TF_VAR_ec2_extra_nix_config`, run `tf-apply` to regenerate the config file, then `nixos-rebuild` to apply it.

## SSH access

The SSH key is auto-generated by the wizard at `~/.ssh/<project>`. To SSH in manually:

```sh
ssh -i ~/.ssh/nix-ml-solo ml@$(cd infra/terraform && tofu output -raw ec2_public_ip)
```

The `ml` user has passwordless sudo. The project directory is at `/home/ml/project`.

## Instance restart

EC2 instances get new public IPs on restart. After restarting an instance:

1. `sync` regenerates the SSH config entry with the new IP
2. `mlflow-open` re-establishes the tunnel
3. `train` or `deploy` handles this automatically on the next run

## MLflow on EC2

MLflow runs as a NixOS systemd service. Data is stored at `/home/ml/mlflow.db` on the EBS volume. The volume persists across instance stops but is destroyed by `teardown`. Run `teardown` (which backs up MLflow) before terminating permanently.

## Monitoring

```sh
status                  # quick overview from local machine
ssh ml@<ec2-ip> "systemctl status mlflow"
ssh ml@<ec2-ip> "journalctl -u mlflow -f"
```
